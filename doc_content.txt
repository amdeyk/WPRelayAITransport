PART A
Product Vision & Core Philosophy
1. What WP Remote Shell Is
WP Remote Shell (WRS) is a complete CLI-driven WordPress development and operations platform. It allows you to build, update, and manage every aspect of a WordPress website — pages, content, membership tiers, email, forms, WooCommerce products, database tables, scheduled tasks — entirely from your local terminal. You never need to log into WordPress Admin, cPanel, or FTP for day-to-day operations.
WRS is designed to be operated by an AI CLI tool (such as Claude CLI) under human supervision. It is also fully usable by a human operator directly. The system is built around the assumption that when AI is driving operations, the infrastructure must be fault-tolerant by design — not by convention.
What WRS replaces
WP Admin for daily operations, FTP/cPanel for file management, phpMyAdmin for DB work, manual plugin configuration for features like membership and email
What WRS is not
A visual page builder, a hosting provider, a backup system, or a WordPress theme. It is an operations and deployment platform.
Who operates it
Primarily: an AI CLI tool (Claude CLI or equivalent) issuing structured commands. Secondarily: you, the human, via direct CLI commands or in supervision of the AI.
Where it lives
GitHub repo (the code) + local machine (the CLI tool + your site files) + WordPress server (the plugin that receives commands)

Repeatable operations
Every operation is idempotent — safe to re-run. The deploy system only pushes what has changed.

Every operation produces structured telemetry. The AI always knows what the server actually did, not just whether the HTTP call succeeded.

The AI cannot change its own operation mode, override the circuit breaker, or reset after a failure. Only the human can.

PART B
System Architecture
3. System Architecture
3.1 The Three-Layer Model
Layer 1 — Secure Transport
HTTPS-only bridge between local machine and WordPress server. Every request is HMAC-SHA256 signed with a time-based salt. Handles auth, IP filtering, rate limiting, and replay protection. This is the v1.0 security foundation — all other layers build on top of it.
Layer 2 — Plugin Engine
A PHP module registry installed as a WordPress plugin. One module per capability domain (content, media, database, members, email, forms, WooCommerce, CPT, cron). Each module exposes REST endpoints. The plugin also handles server-side journaling, telemetry generation, and checkpointing.
Layer 3 — CLI Toolkit
A Python CLI on your local machine. One command namespace per module. Manages the local operation journal, circuit breaker state, checkpoint downloads, AI context feed generation, and state reconciliation. All user-facing commands are here.

3.3 Dual Content Path

For structured content: blog posts, CPT entries (team, FAQs, testimonials), menus, media. Standard WordPress REST API. JSON-format local files. Both tracks coexist on the same site and deploy through the same system.

PART C
Security Model
4. Eight-Layer Security Model
⚠️  This plugin creates a privileged HTTP endpoint on your server. It must be treated with the same seriousness as an SSH private key. All eight layers are mandatory. None may be disabled in production.
Layer
Mechanism
Config Field
What It Blocks
1 — HTTPS only
Plugin rejects all HTTP requests with 403 before any auth check
require_https: true
Plaintext interception, credential sniffing
2 — IP allowlist
Request IP checked against allowed_ips array — before token check
allowed_ips: ['x.x.x.x']
Requests from unknown sources
3 — Token verification
X-WRS-Token header verified against bcrypt hash using constant-time comparison
token_hash in plugin.config.json
Brute force, credential stuffing
4 — HMAC-SHA256 signing
Every request: HMAC(token, endpoint|timestamp|sorted_payload). Verified by plugin.
Enforced in auth.php
Request tampering, man-in-the-middle
5 — Time-salt anti-replay
X-WRS-Time must be within ±replay_window_seconds. Same payload hash rejected twice.
replay_window_seconds: 30
Replay attacks, captured request reuse
6 — Rate limiting
Max rate_limit_per_minute requests per IP. Returns 429 on excess.
rate_limit_per_minute: 20
Brute force, DoS
7 — Command blocklist
Hardcoded block of catastrophic shell patterns (rm -rf /, mkfs, dd if=/dev/zero, etc.)
Hardcoded — not configurable
Accidental or malicious destructive commands
8 — Master toggle
When off, all endpoints return 404. No information leakage.
modules.master_enabled in plugin config
Exposure when not actively using
4.1 Token Architecture
Generation
openssl rand -hex 32 — 64-char hex, 256 bits of entropy. Generated by wizard.
Local storage
Plaintext token in ~/.wrs/sites/<site>/local.config.json — used to sign requests. Never in the GitHub repo.
Server storage
bcrypt hash of the token in plugin.config.json — plugin verifies against hash. Plaintext never reaches the server.
Rotation
python setup/wizard.py --rotate-token — new token generated, both sides updated atomically in one command.
If leaked
Run token rotation immediately. Old hash is replaced on server within seconds.
wp-config.php line
One define() statement: WRS_CONFIG_PATH pointing to plugin.config.json. Only constant in the codebase.
4.2 Configuration Files — Security Rules
plugin.config.json
Lives on server at WRS_CONFIG_PATH. Contains token_hash, allowed_ips, module config. Never in git. Deployed via wrs setup deploy-config.
local.config.json
Lives at ~/.wrs/sites/<site>/local.config.json. Contains plaintext token. Never in git. Never in the project folder.
.gitignore rules
Both config files are gitignored by default. The repo contains only config templates with placeholder values.
config/ directory
Contains .template.json files only — placeholders. Generated files are in ~/.wrs/ (local) and on the server. Never overlap.

PART E
Page Building & Content Management
6. HTML/CSS Page Building
6.1 How It Works
WRS injects HTML directly into WordPress post_content. There is no page builder dependency. The WordPress database stores your HTML as-is, and the active theme renders it inside the theme template. For full-width control, canvas mode bypasses the theme entirely.
Page format
Standard .html file — any valid HTML. One file per page in your project's pages/ folder.
CSS format
Standard .css file. Two modes: inline (CSS prepended as <style> tag in HTML) or enqueue (CSS saved as a file and wp_enqueue_style'd for that page only).
WP storage
post_content = HTML. _wrs_page_mode = 'html' meta prevents other editors from overwriting it.
Shortcode support
WordPress do_shortcode() is applied — PMPro, CF7, WooCommerce shortcodes embedded in HTML work normally.
Canvas mode
wrs page build --canvas — renders with wrs-canvas.php template: only your HTML, no theme header/footer/sidebar. Full visual control.
SEO compatibility
Pages are standard WP pages — Yoast/RankMath SEO meta works. wrs page set-meta writes to their fields.

PART G
Fault Tolerance, Observability & AI Safety
15. The Fault Tolerance Problem
The loop problem: AI pushes an operation → WordPress crashes mid-write → AI retries (assumes nothing was written) → creates duplicate partial state → retries again → now three partially-written records and a broken site. No record of what the AI did or in what order. This is not a theoretical risk — it is the default failure mode of any AI-driven system without fault tolerance built in.
The five components of the fault tolerance layer address this problem systematically. They are not optional features — they are the foundation that makes AI-driven operation safe.
Component 1 — Operation Journal
Append-only log of every operation on both sides. Written before execution, not after.
Component 2 — Server Telemetry
Server reports actual outcome: what it wrote, PHP errors, DB rows affected, memory used, recovery hint.
Component 3 — Checkpoint System
Server-side snapshot of affected state before every write. The rollback target if the operation fails.
Component 4 — AI Circuit Breaker
CLI halts all AI write operations after N consecutive failures. Human must confirm before resuming.
Component 5 — State Reconciliation
Compare local journal view vs actual server state. Detect drift before the next operation.
16. Operation Journal
16.1 Overview
Local journal
~/.wrs/journal/<site-slug>/journal.ndjson — append-only NDJSON. Written with fsync before any HTTP request.
Server journal
wp-content/wrs/journal/journal.ndjson — written by plugin via file_put_contents with LOCK_EX before any DB write. Works even if WP is partially broken.
Format
Newline-delimited JSON. One entry per line. Never edited — only appended. Readable with any text editor or jq.
Rotation
Rotate at 10MB. Keep last 5 rotations. Same policy on both sides.
Git
NOT committed to project git repo. Lives in ~/.wrs/ (local) and wp-content/wrs/ (server).
16.2 Journal Entry Schema
op_id
UUID v4 — unique identifier, same in local and server entries
op_type
page.build, page.update, db.migrate, members.create-tier, etc.
status
PENDING → IN_FLIGHT → SUCCESS / FAILED / PARTIAL / ROLLED_BACK
timestamp_start
ISO 8601 — when entry was first written (before execution)
timestamp_end
ISO 8601 — when operation completed (null if in-flight)
payload_hash
SHA256 of operation payload — detects identical retries
payload_summary
Human-readable: 'page.build slug=home title=Home 4.2KB HTML'

ID of the pre-operation checkpoint (null if no checkpoint)
server_outcome

error_detail
Full error message if FAILED or PARTIAL
retry_of
op_id of original operation if this is a retry — chains retries together
operator
'human' or 'ai'
ai_session_id
AI conversation/session ID if operator = 'ai'
PARTIAL is the most dangerous status. It means the server started writing but did not finish. A blind retry adds more inconsistency on top of partial state. The system surfaces PARTIAL status prominently and blocks AI retries until human reviews.

19. AI Circuit Breaker
The circuit breaker is enforced in the WRS CLI code — not as a suggestion to the AI. The AI cannot bypass it, override it, or reset it. Only a human can reset it.
19.1 States
CLOSED (normal)
All AI operations proceed normally
HALF-OPEN (warning)
Approaching threshold. AI is warned in its context feed. Operations continue.
OPEN (halted)
Threshold exceeded. All AI write operations blocked. CLI returns structured error to AI. Human must intervene.
19.2 Configuration
consecutive_failure_threshold
Failures before OPEN. Default: 3
failure_window_minutes
Only failures within this window count. Default: 10
partial_counts_as_failure
PARTIAL status counts as failure. Default: true
human_pin_to_reset
Human must type confirmation PIN to reset. Default: true
block_reads_when_open
Also block read operations when circuit OPEN. Default: false
19.3 When Circuit Opens
CLI detects N consecutive failures within window
CLI writes CIRCUIT_OPEN event to local journal, notifies server journal
CLI prints prominent red alert in terminal
CLI returns structured error to AI: { circuit_breaker: 'OPEN', reason: '...', human_action_required: true }
All AI write operations blocked — return immediately without executing
Human runs wrs circuit-breaker status — sees full failure history and telemetry
Human fixes the underlying issue
Human runs wrs circuit-breaker reset — enters PIN, circuit returns to CLOSED
19.4 Circuit Breaker Commands
Command
Description

State, consecutive failures, last failure detail, time since last success
wrs circuit-breaker history
Last 10 state changes with triggers
wrs circuit-breaker reset
Reset to CLOSED — requires PIN. Records reset in journal.
wrs circuit-breaker test
Run lightweight health check without writing anything — safe when OPEN
wrs circuit-breaker disable
Disable for session — requires --i-know-what-im-doing flag. Recorded in journal.

Behaviour

23. Recovery Playbooks
23.1 WordPress Crash Recovery
Do NOT retry. Run: wrs circuit-breaker test — minimal health check that works even if WP is broken
Run: wrs server errors — pull PHP error log to understand what crashed WP
wrs journal list --limit 3 — find the op_id of the last operation
If status is PARTIAL: wrs rollback --op-id <id> — restore checkpoint
If server is completely down: wrs rollback --op-id <id> --use-local-checkpoint — generates SQL/file commands for manual restore via cPanel or phpMyAdmin
After recovery: wrs reconcile --all — verify server state matches local journal
wrs circuit-breaker reset
wrs journal show --op-id <id> — review telemetry from the failing operation before retrying
23.2 Plugin Reinstall After Server Wipe
Local journal and checkpoints in ~/.wrs/ survive — they are on your machine
python setup/wizard.py — regenerates plugin ZIP and config files
Upload plugin ZIP to WordPress — the one browser step
wrs setup deploy-config — redeploys plugin.config.json
wrs server health — confirm plugin is running
wrs reconcile --all — everything shows as LOCAL_ONLY (server was wiped)
wrs deploy --force — pushes all local content to the clean server
Your local project folder is the authoritative copy of your site. The WordPress server is the runtime. If the server disappears, redeploy from local. The journal tells you exactly what was on the server before it went down.
23.3 Breaking the AI Retry Loop
Circuit breaker will trigger OPEN after configured consecutive_failure_threshold
wrs journal list --operator ai --status FAILED — see exactly what the AI was attempting
wrs journal show --op-id <id> — check recovery_hint in server_outcome
DO_NOT_RETRY: structural problem (invalid HTML, DB constraint, permission error). Fix root cause, do not retry.
SAFE_TO_RETRY: transient failure (timeout, lock contention). Single manual retry is safe.
wrs circuit-breaker reset — enter PIN to resume
